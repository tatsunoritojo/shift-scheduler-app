# シフリー LP v2 ブリーフ（ClaudeDesign 向け）

**ドラフト**: 2026-04-23 / 作成: 東城 × Claude（PM 補佐）
**目的**: ClaudeDesign に新規 LP を制作してもらうためのブリーフ。デザインの主導権は ClaudeDesign 側にあり、本書は方向性・制約・素材・禁止事項を提供するもの。

---

## 1. プロジェクト背景

- シフリー（Shifree）は、Google カレンダー連携のシフト管理ツール（Flask + Vanilla JS、Vercel 配信）。
- 既存 LP（`static/pages/landing.html`）は 2026-03-17 に「獲得型 LP」として一度リニューアル済み。
- しかし、現状の LP は **AI が作ったとバレバレのデザイン** で、スタートアップ・個人経営事業者に対して信頼感を十分に作れていない。
- 本件は ClaudeDesign の能力テストを兼ねており、**アウトプットの品質次第で既存 LP を置き換える** 判断をする。

---

## 2. ターゲットと勝負の基準

### ターゲット
**スタートアップ、または口コミで広がっていく個人経営レベルの事業所**（飲食・美容・小売・塾・教室など、5〜30 人程度のチーム）。

### 勝負の基準 — 「1 秒で決まる印象」
アクセスした瞬間の 1 秒で次の 2 つを成立させること：

1. **「他とは違いそうだな」** — テンプレートでは見慣れない視覚体験
2. **「これを用意できる人が作るプロダクトなら、信頼できそう」** — craft（作り込み）そのものを信頼の証として使う

LP の出来そのものがプロダクトのプレビュー。これは "作り込みの品質 = プロダクトの品質" というシグナリングであり、ターゲットに対する最大の訴求軸。

---

## 3. デザインの主導権について

**ClaudeDesign にデザインの決定権がある**。本書で指定しているのは次の 3 種：

| 指定の強さ | 内容 |
|---|---|
| 必須（変更不可） | 構成の大枠、AI 臭禁止リスト、技術制約、掲載機能 |
| 推奨（提案可） | 色の方向性、タイポグラフィ、アニメーションの強度 |
| 自由（任意） | 具体的な色番・フォント選定・レイアウトの細部・コピーライティング |

迷ったら「craft signal を最大化する」「既存 LP と被らない」の 2 点を優先基準にしてよい。

## 4. 合意済みの全体構成

```
┌──────────────────────────────────────┐
│ [Hero] 組成アニメ                       │
│   空のグリッド → 希望が入る →             │
│   自動配置 → 完成 を 5-10 秒ループ        │
├──────────────────────────────────────┤
│ [接続部 A] ズームアウト型                 │
├──────────────────────────────────────┤
│ [セクション 1] 1ヶ月体験型                │
│   スクロールで時系列に場面が進む          │
│   月末 → 提出期間 → 締切 → 確定 →          │
│   前日 → 欠員対応                        │
├──────────────────────────────────────┤
│ [接続部 B] ズームアウト型                 │
│   マクロ（1 ヶ月の運用）から              │
│   ミクロ（1 機能の詳細）へ視点が変わる      │
├──────────────────────────────────────┤
│ [セクション 2] 機能深掘り（分割持続型）      │
│   画面を左右非対称に分割                  │
│   片側: プロダクト画面 / 片側: 説明文     │
│   スクロールで両側が同期して進む           │
├──────────────────────────────────────┤
│ [CTA + FAQ + Footer]                  │
└──────────────────────────────────────┘
```

### 採用された craft レバー（設計の軸）

1. **スクロール連動の組成アニメ** — Hero とセクション 1 の両方で機能させる
2. **色の制限** — プロフェッショナル感を演出（詳細は §8）
3. **画面構成のこだわり** — 定型グリッドを崩す、非対称、余白の使い方

### ズームアウト型接続の意図

一般的な LP は「セクション A → セクション B」を見出しと仕切り線で区切る。本 LP では **視点の倍率が変わる** ことで接続する（マクロ→ミクロ）。具体例：

- 1ヶ月体験型の最後の場面が終わった瞬間、画面全体が引いて「同じプロダクトを別の倍率で見直す」演出
- 例: 全場面で共通して映っていた「シフト表の一部」がズームアウトしてアプリ全体の UI に展開される、など

この接続部こそが craft signal の勝負どころ。「1 秒印象」の効果を接続部でも再現する。

## 5. AI 臭アンチパターン禁止リスト

既存 LP に存在するパターンは **すべて禁止**。以下は明確に避けること：

### 色
- `#3b82f6`（Tailwind blue-500、既存 LP のプライマリ）の使用**禁止**
- 紫 → 青、青 → シアンのグラデーション**禁止**
- 定型的な「IT プロダクト色」（#6366f1 インディゴ、#8b5cf6 バイオレット等）**禁止**

### タイポグラフィ・アイコン
- lucide-react 風の線画アイコン（circle-info、calendar、clock 等）の多用**禁止** → 使う場合は独自の表現で置換
- 絵文字の使用**禁止**
- システムフォント（`-apple-system, BlinkMacSystemFont`）だけで済ませる**禁止** → Web フォントを意図的に選ぶ

### コピーライティング
- 挑発型ヒーローコピー（「まだ○○ですか？」「もう○○で悩まない」）**禁止** ← 既存 LP と同じパターン
- 形容詞シリーズ「シンプル、パワフル、美しい、革新的、スマート」**禁止**
- 「次のレベルへ」「ビジネスを加速」「時代に遅れない」等の汎用煽り**禁止**
- 星 5 つの偽社会的証明（実在しないレビュー）**禁止**
- 「〜だけ」「〜秒で」「世界初」等の誇張表現は根拠のあるものだけ可

### レイアウト・構成
- 3 カラム均等の Features グリッド**禁止** → カード 3 枚並べは既存 LP の象徴
- 「悩み提示（Pain）→ 解決（Solution）」のテンプレ二項対立**禁止**（既存 LP がこれ）
- Hero = 中央寄せ大見出し + サブテキスト + 中央 CTA の定型**禁止**
- セクション順「Hero → Features → Pricing → Testimonial → FAQ → CTA」のコピペ**禁止**

### ビジュアル
- ストック写真（ノートパソコンを前にした女性、握手、握り拳、等）**禁止**
- 汎用 3D イラスト（undraw 系、blush.design 系）**禁止**
- グラスモーフィズム（半透明ガラス）の流行踏襲**禁止**
- スマホフレーム型 mockup の使い回し**禁止** ← 既存 LP で使用中

## 6. 1ヶ月体験型セクション — 扱う場面

企業によって締め日は違うが、**月末にかけて翌月のシフトを組む** サイクルが最も一般的。このサイクルを軸に 6 場面を時系列に並べる：

| # | 場面 | Worker 体験 | Admin 体験 | 対応機能 |
|---|---|---|---|---|
| 1 | **月末: 期間公開** | メール/通知で「来月シフト募集」が届く | 期間を作成し、全員に一斉通知 | シフト期間管理、招待システム |
| 2 | **提出期間: 希望集まる** | スマホで出勤可能日をタップして送信 | 提出状況をリアルタイムで確認 | 希望提出、営業時間自動反映 |
| 3 | **締切前: 自動リマインド** | 「あと 1 日です」とメールが届く | 未提出者リストを自動検出 | リマインダー（submission_deadline） |
| 4 | **締切後: シフト作成** | — | 集まった希望をもとにシフトを組み立て | 自動組成、承認プロセス（ON/OFF） |
| 5 | **確定: カレンダー反映** | Google カレンダーにシフトが自動追加 | ワンクリックで確定 → 一斉同期 | Google Calendar 同期 |
| 6 | **前日: 忘れ防止 + 欠員対応** | 前日 21 時にリマインドが届く。もし誰かが急に出られなくなったら、他のスタッフに声がかかる | 欠員時は候補者を自動抽出して通知 | プリシフトリマインド、欠員募集（公平性ソート） |

**演出のポイント**: ズームアウト型接続を意識して、6 場面を通して **何か共通の視覚要素（例: シフト表の同じグリッド）** が画面のどこかに残り続ける構成が望ましい。

## 7. 機能深掘りセクション — 掲載機能リスト

実装済み機能から、LP 掲載の優先度で 8 機能を選定：

| 機能名 | 一言説明 | 差別化ポイント |
|---|---|---|
| **Google カレンダー自動同期** | 確定シフトが各スタッフの Google カレンダーに自動挿入 | スタッフは新アプリ不要 |
| **アプリ不要（PWA）** | ブラウザだけで動く。ホーム画面追加でネイティブ風 | インストール抵抗を排除 |
| **承認プロセス切替** | 個人経営は OFF で即確定、複数人体制は ON で Owner 承認 | 事業規模に合わせて伸縮 |
| **希望の柔軟な提出** | 営業時間に合わせて時間帯単位で可/不可を指定 | 「9-12 のみ OK」等の粒度 |
| **欠員募集の公平性ソート** | 候補者を「週あたり勤務時間が少ない順」で提案 | ベテランに偏らない |
| **招待コード + 個別招待** | QR コードで全員一括招待、個別は role 指定可能 | スタッフ参加の障壁最小化 |
| **自動リマインダー** | 締切前・前日 21 時に自動メール。重複送信防止あり | Admin が個別に連絡不要 |
| **ロール別 UI** | Worker / Owner / Admin で完全に別画面。必要な機能だけ見える | 「どこを触ればいい？」が起きない |

**ClaudeDesign への指示**: これらを 8 個すべて均等に扱う必要はなく、**3〜5 個をメインに濃く、残りをサブ扱い** で差をつけてよい。どれを主役にするかはデザイン判断で決めて OK。

## 8. 色・タイポの方向性（提案）

**決定権は ClaudeDesign 側** だが、参考として 3 方向：

### A. 硬質モノクローム + 1 色
- ベース: 純黒 `#000` / 純白 `#fff` / 微細なグレー数段
- アクセント 1 色のみ（蛍光グリーン、オレンジ、赤、黄のいずれか — blue 系以外）
- 参考: Linear、Raycast、Arc
- 効果: プロフェッショナル感、テック寄り、「作り込みが効く」

### B. ウォームオフホワイト + 深色
- ベース: オフホワイト `#FAFAF5` や `#F5F1E8` 等の有彩 off-white
- アクセント: 深い赤茶 `#8B2E13`、深緑 `#1C3A2E`、濃紺 `#0A1D3A` 等
- 参考: エディトリアル・国内デザイン事務所の自社 LP
- 効果: 人間味、読み物的、「職人感」

### C. 純粋モノクローム（色を排除）
- ベース: 黒・白・グレーのみ。色を一切使わない
- 動きとタイポと余白だけで差別化
- 参考: 一部の建築事務所 LP、写真家ポートフォリオ
- 効果: 最も難易度高く、最も craft signal 強い

**既存 LP との差別化が最大の方向** を優先してよい。

### タイポグラフィ

- **和文**: ゴシック（Noto Sans JP / 游ゴシック系）ではなく、**明朝体** または **モダン日本語フォント**（源ノ明朝、Zen 系、Shippori Mincho）を軸に検討してほしい
- **欧文**: Inter / Helvetica の定型を避け、Neue Haas Grotesk / Söhne / Migra 等の差別化フォントを検討
- 可変フォント（Variable Font）の活用歓迎

## 9. Hero のキャッチコピー（方向性のみ）

**具体的な文言は ClaudeDesign 提案で OK**。以下は避けること：

- 既存 LP「毎月のシフト管理、まだ手作業ですか？」のような挑発/問いかけ型
- 「シフト管理をシンプルに」のような抽象形容詞

許容される方向：
- **観察的短文**（例: 「月末が、少しだけ早く終わる」）
- **機能の静かな名指し**（例: 「出勤希望、自動集約、自動配置、自動通知」）
- **比喩**（例: 「1 ヶ月を、約 30 秒で」← Hero アニメの尺と掛ける）
- **数字の具体**（例: 「スタッフ 12 名・7 ヶ月運用」※実績が育つまで保留可）

**注意**: 実運用データはまだ育っていないので、「累計ユーザー数」「導入社数」のような数字は出せない。この制約の中で信頼感を作る必要がある → 作り込みで勝負する（§2 の勝負基準に戻る）。

## 10. 技術制約

### 配信
- ルート: `/lp`（既存）または `/lp-v2` のどちらでも可
- 配信: Flask の `send_static_file`（`api_common.py` 経由）
- 既存 LP は `static/pages/landing.html` なので、v2 は `static/pages/landing-v2.html` として並存

### ファイル構成
- HTML: `static/pages/landing-v2.html`
- CSS: `static/css/landing-v2.css`（`common.css` との共存は OK だが、変数名の衝突は避ける）
- JS: `static/js/landing-v2.js`（必要なら）
- アニメーション: CSS アニメーション優先、複雑な場合は Canvas / SVG / WebGL も可
- 動画: MP4 使用可。軽量化必須（目安 1-2MB 以内、ポスター画像は別）

### CSP・セキュリティ
- インラインスクリプトは `nonce` または外部ファイル化
- 外部 CDN 依存は最小限（フォントは Google Fonts OK）
- 画像は WebP 推奨、遅延読込（`loading="lazy"`）

### SEO・アクセシビリティ
- 既存 LP の meta / OG / JSON-LD を継承（`schema.org SoftwareApplication`、FAQ）
- Skip link、セマンティック HTML、適切な heading 階層
- `prefers-reduced-motion` 対応（Hero アニメは停止可能に）
- `aria-label` / `alt` 属性は適切に

### パフォーマンス
- LCP 2.5s 以内、CLS 0.1 以内を目標
- Hero アニメは軽量優先、重いなら遅延実行

### 既存アセット（流用可）
- ロゴ: `/static/icons/icon-192.png`、`/static/icons/favicon-32.png`
- manifest: `/static/manifest.json`
- 既存 mockup 用 CSS（`mock-*` クラス群）は参考にしてよいが、そのまま流用は非推奨

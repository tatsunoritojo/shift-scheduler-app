# 技術改善レポート: Phase 1〜5

**対象**: シフリー（Shifree）— シフト管理Webアプリケーション
**期間**: 2026年3月
**読者想定**: 基礎的なIT知識があるジュニアエンジニア

---

## はじめに — このレポートの目的

シフリーは、飲食店や塾など小規模店舗向けのシフト管理アプリです。ワーカーがGoogleカレンダーから空き時間を自動計算してシフト希望を提出し、管理者がシフト表を作成、オーナーが承認する——という一連のワークフローをWebブラウザ上で完結させます。

本レポートでは、このアプリに対して実施した5段階の技術改善（Phase 1〜5）について、**それぞれ何が問題だったのか**、そして**なぜその改善が必要だったのか**を解説します。

改善前のアプリは「動く」状態ではありました。しかし、実際にチームで運用し続ける上で、多くの技術的な課題を抱えていました。

---

## 改善前の状態 — 何が問題だったのか

改善前のアプリ（v1）は、1つのPythonファイルに全機能が詰め込まれた約360行のコードでした。個人開発の初期段階としては自然な状態ですが、以下のような課題がありました。

| 領域 | 問題 | 影響 |
|------|------|------|
| テスト | 自動テストが1件もない | コード変更のたびに手動で全機能を確認する必要がある |
| データベース | スキーマ変更を手動で管理 | 本番DBの更新時にデータが消える危険がある |
| エラー処理 | エラーの返し方がバラバラ | フロントエンドで統一的なエラー処理ができない |
| 権限管理 | 環境変数でメールアドレスを直接指定 | ユーザーの追加・権限変更にサーバーの再起動が必要 |
| セキュリティ | 組織に属さないユーザーもAPIにアクセス可能 | 他人のシフト情報が見えてしまう可能性がある |
| 運用監視 | バックグラウンド処理の成否が見えない | メール未送達やカレンダー同期失敗に気づけない |
| 監査 | 「誰が何をしたか」の記録がない | 不正操作やミスの原因を追跡できない |
| UX | OAuth認証の期限切れ時に無言でエラー | ユーザーが何をすればいいかわからない |

これらは、いずれも「今すぐアプリが壊れる」類の問題ではありません。しかし、**運用を続けるほど深刻化する構造的な問題**です。以下、各Phaseで何をなぜ改善したのかを説明します。

## Phase 1: 自動テスト・DBマイグレーション・エラー標準化

### なぜテストが必要なのか

ソフトウェア開発では、コードを変更するたびに「既存の機能が壊れていないか」を確認する必要があります。テストがない状態では、この確認をすべて人間が手動で行います。

具体的な問題を挙げます。

- ログイン機能を修正したとき、シフト提出機能が壊れていないか確認しましたか？
- APIのレスポンス形式を変えたとき、フロントエンドの全画面で正しく動作しますか？
- 日付のバリデーションを直したとき、営業時間の登録に影響は出ていませんか？

機能が20個あれば、確認項目は簡単に100を超えます。これを毎回手動で行うのは現実的ではなく、確認漏れからバグが本番環境に入り込みます。

**改善内容**: pytestによる自動テスト基盤を構築し、テスト用の共通フィクスチャ（テストデータの自動生成・ログイン模倣ヘルパーなど）を整備しました。

```
tests/
├── conftest.py          # テストの共通設定（DB初期化、ユーザー生成、認証ヘルパー）
├── test_auth.py         # 認証テスト（20件）
├── test_approval.py     # 承認ワークフローテスト（10件）
├── test_validation.py   # 入力検証テスト（38件）
└── ...                  # 最終的に153件
```

`python -m pytest tests/` を1回実行するだけで、数秒で全機能の正常性を検証できます。

### なぜDBマイグレーションが必要なのか

アプリの開発が進むと、データベースのテーブル構造（スキーマ）を変更する場面が必ず出てきます。例えば「監査ログ用のテーブルを追加したい」「ユーザーテーブルに新しいカラムを追加したい」といったケースです。

改善前は `db.create_all()` という関数でテーブルを作っていました。これは**テーブルが存在しなければ作る**という動作ですが、**既存テーブルの変更は一切行いません**。つまり、カラムを追加したい場合は、テーブルを削除して作り直す必要がありました。本番環境でこれを行うと、既存データがすべて消えます。

**改善内容**: Flask-Migrate（Alembic）を導入し、スキーマの変更履歴をコードとして管理するようにしました。

```
migrations/versions/
├── 25086bfdac9f_initial.py                    # 初期スキーマ（13テーブル）
├── dc3fa46ab193_add_organization_members.py   # 組織メンバーテーブル追加
├── a1b2c3d4e5f6_add_async_tasks.py            # 非同期タスクテーブル追加
└── b2c3d4e5f6a7_add_audit_logs.py             # 監査ログテーブル追加
```

各ファイルには `upgrade()`（適用）と `downgrade()`（取り消し）の関数があり、`flask db upgrade` を実行するだけで、現在のDBに必要な変更だけが安全に適用されます。本番環境のデータを失う心配がなくなりました。

### なぜエラー形式の標準化が必要なのか

改善前のAPIは、エラーが発生したとき、それぞれの場所で自由な形式でレスポンスを返していました。

```json
// ある場所では
{"message": "not found"}

// 別の場所では
{"error": "認証が必要です"}

// さらに別の場所では
"Bad Request"
```

フロントエンド側でエラーを処理するコードは、この3パターンすべてに対応する必要があります。新しいAPIを追加するたびに、また別のパターンが増えるかもしれません。

**改善内容**: `APIError` 例外クラスと `error_response()` ヘルパーを作成し、全APIで統一されたエラー形式を強制しました。

```json
{
  "error": "認証が必要です",
  "code": "AUTH_REQUIRED"
}
```

`code` フィールドは機械が読むための識別子です。フロントエンドは `code` を見て処理を分岐できます。人間向けのメッセージ（`error`）は日本語でも英語でも自由に変えられますが、`code` は変わらないため、フロントエンドのコードを修正する必要がありません。

## Phase 2: 権限管理のDB化と招待制

### なぜ環境変数ベースの権限管理では不十分なのか

改善前の権限管理は、次のような仕組みでした。

```
環境変数 ADMIN_EMAIL=admin@example.com
環境変数 OWNER_EMAIL=owner@example.com
→ 上記以外のメールアドレスは全員「ワーカー」
```

この方式には、深刻な運用上の問題があります。

1. **管理者を追加するたびにサーバーを再起動する必要がある**。環境変数はアプリ起動時に読み込まれるため、変更するには再起動（=一時的なサービス停止）が必要です。
2. **権限変更の履歴が残らない**。誰がいつ管理者になったのか追跡できません。
3. **複数の組織（店舗）を管理できない**。1つの環境変数に1セットの管理者しか設定できないため、「A店の管理者」と「B店の管理者」を分けることが原理的に不可能です。
4. **誰でもログインすれば「ワーカー」になれてしまう**。Googleアカウントを持っていれば、招待されていなくてもアプリにアクセスできてしまいます。

**改善内容**: `OrganizationMember` モデルをデータベースに作成し、権限管理をDB駆動に移行しました。

```
Organization（組織）
  └── OrganizationMember（メンバーシップ）
        ├── user_id     → どのユーザーが
        ├── org_id      → どの組織に
        ├── role        → どの権限で（admin / owner / worker）
        └── is_active   → 有効かどうか
```

権限の変更はAPIを通じてDBを更新するだけで即座に反映され、サーバーの再起動は不要です。

### なぜ招待制が必要なのか

権限管理をDB化しただけでは、「誰でもログインすればワーカーになれる」問題は解決しません。そこで、**招待トークン**による参加制御を導入しました。

```
管理者: 招待リンクを生成（有効期限7日、メールアドレス制限可）
    ↓
ワーカー: 招待リンク経由でログイン
    ↓
システム: トークンを検証 → 組織に自動参加 → 指定ロールを付与
```

招待リンクを持たないユーザーがログインしても、組織に所属できないため、業務データにアクセスすることはできません。これにより、意図しないユーザーの参加を防ぎます。

## Phase 3: 非同期タスクキューと運用ダッシュボード

### なぜ同期処理では問題があるのか

シフトが確定すると、ワーカーのGoogleカレンダーにイベントを自動登録する機能があります。改善前は、この処理をAPIリクエストの中で**同期的に**（=その場で完了を待って）実行していました。

```
ユーザー操作: 「確定」ボタンを押す
    ↓  APIリクエスト送信
サーバー: シフトをDBに保存
    ↓  Google Calendar APIを10人分呼び出し（各1〜3秒）
    ↓  メール通知を送信（SMTP接続に2〜5秒）
    ↓  合計: 15〜35秒
ユーザー: ようやくレスポンスが返ってくる（もしくはタイムアウト）
```

ユーザーは「確定」ボタンを押してから30秒近く待たされることになります。さらに、途中でGoogle APIがエラーを返した場合、5人目まで成功して6人目で失敗、という中途半端な状態になりかねません。

また、本アプリはVercelのサーバーレス環境で動いています。サーバーレス関数には**実行時間の上限**（通常10〜60秒）があるため、処理が長引くとタイムアウトで強制終了されます。

**改善内容**: `AsyncTask` モデルによるタスクキューを導入しました。

```
ユーザー操作: 「確定」ボタンを押す
    ↓  APIリクエスト送信
サーバー: シフトをDBに保存 + タスクをキューに登録
    ↓  即座にレスポンス返却（0.5秒以内）
ユーザー: すぐに画面が更新される

--- 数分後 ---

Vercel Cron: /api/cron/process-tasks を呼び出し
タスクランナー: キューからタスクを取り出して実行
    ├── カレンダー同期（成功 → completed）
    ├── メール送信（成功 → completed）
    └── 失敗したタスク → 指数バックオフでリトライ
```

**指数バックオフ**とは、リトライの間隔を段階的に広げる仕組みです。1回目の失敗後は30秒待ち、2回目は2分、3回目は8分と間隔を広げます。これにより、一時的な障害（Google APIの一時停止など）が回復する時間を確保しつつ、サーバーに過剰な負荷をかけることを防ぎます。最大3回リトライしても失敗した場合は `dead`（処理不能）としてマークされ、管理者がダッシュボードで確認できます。

### なぜ運用ダッシュボードが必要なのか

非同期処理を導入すると、「メールは送れたのか」「カレンダー同期は成功したのか」がユーザー操作の時点ではわかりません。問題が起きたとき、管理者がシステムの状態を確認する手段が必要です。

**改善内容**: 管理者向けの運用ダッシュボードAPIを作成しました。

```
/api/admin/dashboard/overview     → 全体サマリー（直近24時間のタスク成功率など）
/api/admin/dashboard/tasks        → タスク一覧（ステータスでフィルタ可能）
/api/admin/dashboard/task-stats   → タスク種別ごとの成功/失敗率
```

これにより、管理者は「メール通知の送信成功率が低下している」「カレンダー同期が特定ユーザーで失敗し続けている」といった問題を、サーバーのログを直接見ることなく把握できます。

## Phase 4: マルチテナント強制・CORS強化・フロントエンド整理

### なぜ「組織メンバーシップの強制」が必要なのか

Phase 2で招待制を導入しましたが、それだけでは不十分でした。組織に所属していないユーザーが、直接APIのURLを叩けばデータにアクセスできてしまう状態が残っていたからです。

```
例: ブラウザのアドレスバーに直接
    /api/admin/members
    と入力すると、認証さえ通っていればデータが返ってしまう
```

これは「認証（あなたは誰か）」と「認可（あなたに何が許可されているか）」の違いに関わる問題です。ログインしていること（認証済み）と、特定の組織のデータにアクセスする権利があること（認可済み）は別の話です。

**改善内容**: 認証ミドルウェア（`@require_auth`, `@require_role`）に、組織メンバーシップの確認を追加しました。

```python
# 改善前: ログインしていればOK
@require_auth  # → session にuser_idがあれば通過

# 改善後: ログイン + 有効な組織メンバーシップが必要
@require_auth  # → session にuser_idがある + OrganizationMemberが存在する
```

組織に所属していないユーザーがAPIにアクセスすると、`403 ORG_MEMBERSHIP_REQUIRED` エラーが返り、専用の案内ページ（`/no-organization`）にリダイレクトされます。

### なぜCORSの「フェイルクローズ」が必要なのか

**CORS**（Cross-Origin Resource Sharing）は、「どのWebサイトからのリクエストを受け付けるか」を制御するブラウザのセキュリティ機構です。

例えば、`https://evil-site.com` 上のJavaScriptから `https://shifree.vercel.app/api/admin/members` へリクエストを送ることを想像してください。CORSが適切に設定されていなければ、悪意のあるサイトがユーザーのブラウザを経由して、シフリーのAPIを不正に呼び出すことが可能になります。

改善前のCORS設定には「設定忘れ」のリスクがありました。

```python
# 改善前: 環境変数が未設定の場合、何が起きるか不明確
origins = os.environ.get('CORS_ALLOWED_ORIGINS', '*')  # '*' = 全オリジン許可？

# 改善後: 未設定の場合は「何も許可しない」（フェイルクローズ）
origins = os.environ.get('CORS_ALLOWED_ORIGINS', '').split(',')
# 空文字列 → 空リスト → どのオリジンからのリクエストも拒否
```

**フェイルクローズ**とは、「設定が不完全な場合は、安全な側（拒否）に倒す」という設計原則です。反対の「フェイルオープン」（設定がなければ全許可）は、設定忘れが即座にセキュリティホールになります。

### なぜフロントエンドの共通モジュール化が必要なのか

シフリーには、ロール別に3つのJavaScriptファイルがあります（`admin-app.js`, `worker-app.js`, `owner-app.js`）。改善前は、同じ機能のコードが3ファイルに重複していました。

```javascript
// admin-app.js に showConfirmDialog() が定義されている
// worker-app.js にも同じ showConfirmDialog() が定義されている
// owner-app.js にも同じ showConfirmDialog() が定義されている
```

この状態で「確認ダイアログのデザインを変更したい」となったとき、3箇所すべてを同じように修正する必要があります。1箇所でも修正漏れがあれば、ロールによって挙動が異なるバグが生まれます。

**改善内容**: 共通関数を独立したモジュールに切り出しました。

```
static/js/modules/
├── api-client.js      # API通信（認証エラー処理含む）
├── ui-dialogs.js      # 確認ダイアログ（showConfirmDialog）
└── event-utils.js     # カレンダーイベント処理（日付フィルタ、表示整形）
```

各ロール別ファイルは `import { showConfirmDialog } from './modules/ui-dialogs.js'` として共通モジュールを使います。修正は1箇所で済み、全ロールに自動的に反映されます。

### なぜCSP（Content Security Policy）が重要なのか

**CSP**は、Webページ内で実行できるスクリプトやリソースの読み込み元を制限するセキュリティ機構です。

もしアプリにXSS（クロスサイトスクリプティング）脆弱性が存在した場合、攻撃者はページ内に悪意のあるスクリプトを注入できます。CSPが設定されていれば、たとえ脆弱性があっても、許可されていないスクリプトの実行をブラウザが阻止します。

```
Content-Security-Policy: script-src 'self' https://unpkg.com
```

この設定は「スクリプトは自サイト（`'self'`）と `unpkg.com` からのみ実行を許可する」という意味です。攻撃者が `<script>alert('hacked')</script>` をページに注入しても、インラインスクリプトは許可リストにないため、ブラウザが実行を拒否します。

実際に、Phase 4の初期実装で `no-organization.html` にインラインスクリプトを書いてしまい、CSPに違反していることがレビューで発覚しました。スクリプトを外部ファイル（`no-org.js`）に切り出すことで修正しました。CSPは、開発者自身のうっかりミスからもアプリを守ります。

## Phase 5: 監査ログとOAuth再認証UX

### なぜ監査ログが必要なのか

監査ログとは、「誰が」「いつ」「何をしたか」の記録です。一見、ログファイル（`print` や `logging`）で十分に思えるかもしれませんが、両者には重要な違いがあります。

| 比較項目 | アプリケーションログ | 監査ログ |
|----------|---------------------|----------|
| 目的 | デバッグ・障害調査 | 責任追跡・不正検知 |
| 保存場所 | ログファイル（揮発性） | データベース（永続・検索可能） |
| 内容 | 任意のテキスト | 構造化データ（誰・何・いつ・変更前後） |
| 閲覧者 | 開発者 | 管理者（ダッシュボード経由） |

具体例を挙げます。「あるワーカーの権限が、いつの間にかオーナーに変わっている」という問題が発生したとします。

- **監査ログがない場合**: サーバーのログファイルを時間をかけて検索する必要があり、ログがローテーション（定期削除）されていれば証跡がなくなっている可能性もあります。
- **監査ログがある場合**: ダッシュボードで `action=ROLE_CHANGED` で検索するだけで、「3月1日 14:23に admin@example.com が worker→owner に変更した」という記録が即座に見つかります。

**改善内容**: `AuditLog` モデルとAPI、そしてセキュリティ上重要な全操作へのログ記録を追加しました。

```
記録される操作:
├── ROLE_CHANGED         — 権限の変更
├── MEMBER_REMOVED       — メンバーの除外
├── INVITATION_CREATED   — 招待リンクの作成
├── INVITATION_REVOKED   — 招待リンクの失効
├── SCHEDULE_SUBMITTED   — シフト表の承認申請
├── SCHEDULE_APPROVED    — シフト表の承認
├── SCHEDULE_REJECTED    — シフト表の差し戻し
└── SCHEDULE_CONFIRMED   — シフト表の確定
```

各ログには、操作者（`actor_id`）、対象リソース、変更前の値（`old_values`）、変更後の値（`new_values`）、IPアドレス、成功/失敗の区別が記録されます。

### なぜOAuth再認証のUX改善が必要なのか

シフリーはGoogleカレンダーと連携するために、ユーザーのGoogleアカウントに対するアクセストークンを保持しています。このトークンには有効期限があり、以下の場合に無効化されます。

- 一定期間（通常6ヶ月）使用されなかった場合
- ユーザーがGoogleアカウントの設定で「アクセスを取り消し」た場合
- Googleのセキュリティポリシーにより強制失効された場合

改善前は、トークンが無効化された場合の処理が不十分でした。

```
改善前の挙動:
ユーザー: カレンダー画面を開く
    ↓
サーバー: Googleにトークンのリフレッシュを要求 → 拒否される
    ↓
サーバー: 500 Internal Server Error を返す
    ↓
ユーザー: 「エラーが発生しました」という表示を見て、何をすればいいかわからない
```

ユーザーにとって、「再ログインすれば直る」のか「アプリ側の障害なのか」の区別がつきません。

**改善内容**: トークン失効を専用の例外（`CredentialsExpiredError`）として識別し、フロントエンドで再認証モーダルを表示するようにしました。

```
改善後の挙動:
ユーザー: カレンダー画面を開く
    ↓
サーバー: Googleにトークンのリフレッシュを要求 → 拒否される
    ↓
サーバー: 失効したトークンをDBから削除
    ↓
サーバー: 401 CREDENTIALS_EXPIRED を返す
    ↓
フロントエンド: 「再認証が必要です — Googleで再ログイン」というモーダルを表示
    ↓
ユーザー: ボタンを押して再ログイン → 新しいトークンが発行されて復旧
```

ポイントは2つあります。

1. **失効したトークンをDBから即座に削除する**。削除しないと、次のリクエストでも同じエラーが繰り返され、Googleへの無駄なリクエストが発生し続けます。
2. **`CREDENTIALS_EXPIRED` というエラーコードを返す**。Phase 1で標準化したエラー形式が、ここで活きています。フロントエンドは `code` フィールドを見て、通常の認証エラー（ログインページへリダイレクト）か、トークン失効（再認証モーダルを表示）かを判別できます。

## 改善の全体像

以下の図は、5つのPhaseが「アプリの品質」のどの側面に影響しているかを示しています。

```
                    ┌──────────────────────────────────────────┐
                    │            アプリの品質                     │
                    └──────────────────────────────────────────┘
                       │         │          │          │
              ┌────────┘    ┌────┘     ┌────┘     ┌────┘
              ▼             ▼          ▼          ▼
          信頼性         安全性      運用性      保守性
              │             │          │          │
    Phase 1 ──┤   Phase 4 ──┤  Phase 3 ─┤  Phase 1 ─┤
    (テスト)   │   (CORS)    │  (非同期)  │  (テスト)  │
              │             │          │          │
    Phase 3 ──┤   Phase 2 ──┤  Phase 5 ─┤  Phase 4 ─┤
    (リトライ) │   (招待制)   │  (監査)   │  (モジュール化)
              │             │          │          │
    Phase 1 ──┘   Phase 4 ──┤          │  Phase 1 ─┘
    (エラー標準化)  (CSP)     │          │  (マイグレーション)
                            │          │
              Phase 5 ──────┘  Phase 5 ─┘
              (再認証UX)        (ダッシュボード)
```

各Phaseは独立した改善ではなく、相互に関連しています。

- Phase 1の**エラー標準化**がなければ、Phase 5の `CREDENTIALS_EXPIRED` による再認証モーダルは実現できません。
- Phase 2の**DB権限管理**がなければ、Phase 4の**マルチテナント強制**は実装できません。
- Phase 1の**マイグレーション**がなければ、Phase 3の `async_tasks` テーブルも Phase 5の `audit_logs` テーブルも安全に追加できません。

このように、Phase 1〜5は**積み上げ式の設計**になっており、前のPhaseが次のPhaseの基盤となっています。

## まとめ

Phase 1〜5の改善は、すべて**「動くアプリ」を「運用できるアプリ」にする**ための作業でした。

個人開発やプロトタイプの段階では、テストがなくても、権限管理が簡易でも、エラー処理が雑でも、アプリは動きます。しかし、複数のユーザーが日常的に使うアプリでは、次のような問いに答える必要があります。

- コードを変更しても壊れていないと**証明**できるか？ → Phase 1（テスト）
- DBスキーマの変更で本番データが消えないと**保証**できるか？ → Phase 1（マイグレーション）
- 権限の変更を**安全かつ即座に**行えるか？ → Phase 2（RBAC DB化）
- 招待されていない人のアクセスを**確実に遮断**できるか？ → Phase 2, 4（招待制 + マルチテナント強制）
- ユーザーを待たせず、かつ失敗しても**自動で復旧**できるか？ → Phase 3（非同期 + リトライ）
- 問題が起きたときに**原因を特定**できるか？ → Phase 3, 5（ダッシュボード + 監査ログ）
- 設定ミスがあっても**安全側に倒れる**か？ → Phase 4（CORS, CSP）
- エラーが起きたとき、ユーザーが**自力で復旧**できるか？ → Phase 5（再認証UX）

これらの問いに「はい」と答えられることが、プロダクション品質のソフトウェアの条件です。

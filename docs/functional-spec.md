## 2. 画面別機能詳細

### 2.1 管理者ダッシュボード (admin.html)

4タブ構成:

#### タブ1: 設定
| 機能 | 説明 |
|---|---|
| 営業時間設定 | 曜日別の営業開始・終了時刻、定休日指定 |
| 例外日設定 | 祝日・臨時休業・特別営業の日付指定 |
| カレンダー同期 | Google Calendar → 例外日インポート / 営業時間 → Calendar エクスポート |
| リマインダー設定 | 提出期限前の通知日数・時刻、シフト前日通知 ON/OFF |

#### タブ2: 期間
| 機能 | 説明 |
|---|---|
| 期間作成 | 名前・開始日・終了日・提出期限を指定 |
| 期間公開 | draft → open (スタッフに希望提出が可能になる) |
| 提出状況確認 | 各スタッフの提出有無・提出日時 |
| 手動リマインド | 未提出スタッフへ一括リマインダー送信 |

#### タブ3: シフト表 (ビルダー)
| 機能 | 説明 |
|---|---|
| カレンダー表示 | 月間カレンダーに充足状況を色分け表示 (緑=充足、黄=一部、青=未割当、灰=休業) |
| 日別ポップアップ | 日付クリックで営業時間・カバレッジタイムライン・スタッフ割当トグル表示 |
| 提出状況サイドバー | 全スタッフの提出進捗バー |
| 勤務時間サイドバー | スタッフ別の月間合計時間バーチャート |
| 承認依頼 | 構築完了後、事業主へ承認依頼を送信 |
| 確定 | 承認後、カレンダー自動同期を実行 |

#### タブ4: メンバー
| 機能 | 説明 |
|---|---|
| メンバー一覧 | 名前・メール・ロール・参加日表示 |
| ロール変更 | admin / owner / worker を切替 (最後の admin は削除不可) |
| メンバー削除 | 組織からの除外 |
| 個別招待 | メールアドレス + ロール指定で招待トークン発行・メール送信 |
| 招待コード | QR コード + URL で組織への招待リンク生成・有効/無効切替 |

## 3. 状態遷移図

### 3.1 シフト期間 (ShiftPeriod.status)

```
draft ──→ open ──→ closed ──→ finalized
           │
           └── (手動で閉鎖可能)
```

| 状態 | 意味 | 遷移条件 |
|---|---|---|
| draft | 下書き | 作成直後 |
| open | 募集中 | 管理者が公開操作 |
| closed | 募集終了 | 期限到来 or 管理者が手動閉鎖 |
| finalized | 確定済み | スケジュール確定後 |

### 3.2 スケジュール (ShiftSchedule.status)

```
draft ──→ pending_approval ──→ approved ──→ confirmed
                    │
                    └──→ rejected ──→ draft (再構築)
```

| 状態 | 意味 | 遷移条件 |
|---|---|---|
| draft | 構築中 | 管理者がスケジュール保存 |
| pending_approval | 承認待ち | 管理者が承認依頼 |
| approved | 承認済み | 事業主が承認 |
| rejected | 差戻し | 事業主が差戻し |
| confirmed | 確定 | 管理者が確定 → カレンダー同期 |

### 3.3 欠員リクエスト (VacancyRequest.status)

```
open ──→ notified ──→ accepted
              │
              └──→ expired / cancelled
```

| 状態 | 意味 |
|---|---|
| open | 作成済み |
| notified | 候補者に通知済み |
| accepted | 候補者が承諾 |
| expired | 有効期限切れ |
| cancelled | 管理者がキャンセル |

### 3.4 非同期タスク (AsyncTask.status)

```
pending ──→ running ──→ completed
                │
                └──→ failed ──→ (retry) pending
                        │
                        └──→ dead (max_retries 到達)
```

## 4. ビジネスルール

### 4.1 バリデーション

| ルール | 内容 |
|---|---|
| 期間の日付範囲 | start_date < end_date、最大366日 |
| 提出期限 | start_date 以降、end_date 以前 |
| 営業時間 | start_time < end_time (HH:MM 形式) |
| テキスト長 | 期間名: 200文字、備考: 2000文字 |
| スケジュールエントリ | shift_date が期間範囲内、user がアクティブメンバー |

### 4.2 権限ルール

| ルール | 内容 |
|---|---|
| 最後の admin 保護 | 組織に admin が1人の場合、そのメンバーの削除・ロール変更は不可 |
| 組織隔離 | 全 API で user.organization_id == resource.organization_id を強制 |
| 招待トークンメール制限 | email 指定の招待は、そのメールアドレスでのみ使用可 |
| 招待トークン有効期限 | デフォルト7日、使用済み or 期限切れのトークンは無効 |

### 4.3 欠員補充ルール

| ルール | 内容 |
|---|---|
| 候補者抽出 | 対象日に is_available=True かつシフト未割当のアクティブメンバー |
| 公平性ソート | 週間勤務時間の少ない順に並替 |
| 先着受理 | 複数候補者が同時に承諾してもトランザクション制御で1名のみ |
| レースコンディション防止 | Compare-and-swap: 既に accepted なら後発は拒否 |

## 5. 通知一覧

| トリガー | 送信先 | 件名パターン | 送信方式 |
|---|---|---|---|
| 管理者がスケジュール承認依頼 | 事業主 | 承認依頼: {期間名} | 非同期 (AsyncTask) |
| 事業主が承認 or 差戻し | 管理者 | 承認/差戻し: {期間名} | 非同期 |
| 管理者がメール指定招待作成 | 招待先 | {組織名} への招待 | 非同期 |
| 提出期限前 (自動) | 未提出スタッフ | 提出期限リマインド: {期間名} | Cron → 非同期 |
| シフト前日 (自動) | 翌日シフトのスタッフ | シフトリマインド: {日付} | Cron → 非同期 |
| 管理者が欠員通知送信 | 候補者 | 欠員補充のお願い: {日付} | 非同期 |
| 候補者が欠員承諾 | 管理者 | 欠員補充完了: {日付} | 非同期 |

## 6. 認証・参加フロー

### 6.1 初回ログイン (招待あり)

```
1. 管理者が招待リンク or QRコードを共有
2. 新メンバーがリンクアクセス → invite.html 表示
3. 「Googleでログイン」クリック → OAuth フロー
4. OAuth コールバック: 招待トークン検証 → 組織参加 + ロール割当
5. ロール別ダッシュボードへリダイレクト
```

### 6.2 初回ログイン (招待なし)

```
1. ユーザーが /login からログイン
2. OAuth コールバック: 招待トークンなし
3. ADMIN_EMAIL / OWNER_EMAIL 環境変数に一致 → ブートストラップ参加
4. 一致しない → /no-organization ページへ
5. 組織作成 or 招待コード入力で参加
```

### 6.3 Google Calendar 連携

```
1. 初回 OAuth 時に calendar スコープを取得
2. refresh_token を Fernet 暗号化して DB 保存
3. カレンダー参照時: refresh_token → access_token 自動更新
4. ユーザーが Google 側で連携解除 → RefreshError → 再認証モーダル表示
```

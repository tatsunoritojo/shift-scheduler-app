# Shifree UX改善ロードマップ v2

策定日: 2026-03-17

## 設計原則

### 通知チャネルルール

| チャネル | 用途 | 例 |
|---------|------|-----|
| **ページ上部バナー** | その画面で継続する制約・前提条件不足 | 「営業時間が未設定です」 |
| **インラインエラー** | 特定フォーム・項目の問題 | 日付フィールド下の赤字 |
| **トースト** | 一時的な操作結果通知 | 「保存しました」「同期に失敗しました」 |
| **モーダル** | 意思決定が必要なとき限定 | 確認ダイアログ、再認証要求 |

原則: バナーとインラインは画面文脈、トーストは一過性、モーダルは意思決定。混在させない。

### 画面共通状態ルール

以下のルールは全画面で統一的に適用する。画面ごとの手書き実装ではなく、共通ガード関数で制御する。

| 状態 | ルール |
|------|--------|
| `period.status !== 'open'` | 編集・提出操作を不可にする |
| `credentials === missing/expired/revoked` | 同期系操作を不可にする（disabled + 説明テキスト + CTA） |
| `loading === true` | 破壊的操作ボタンを disabled にする |
| `dirty === true` | 離脱時に確認ダイアログを出す |

### disabled 要素の三段構え

ツールチップだけでは説明にならない（モバイルで効かない）。以下を必ずセットで実装する:

1. **disabled 状態** — ボタン/フォームを操作不可に
2. **補足テキスト** — disabled の理由をインラインで表示
3. **CTA または説明リンク** — 解消するための次のアクション

### エラーハンドリングの責務分離

```
┌─────────────────────────────────────────┐
│  共通層 (api-client.js)                  │
│  - HTTP status → エラーコードマッピング   │
│  - 401 → reauth モーダル表示             │
│  - 429 → レートリミットメッセージ         │
│  - TypeError → ネットワークエラー         │
│  - エラーオブジェクトを throw             │
└──────────────┬──────────────────────────┘
               │ catch
┌──────────────▼──────────────────────────┐
│  画面層 (admin-app.js 等)                │
│  - エラーコードに応じた文脈メッセージ     │
│  - 「提出前データは復元できます」         │
│  - 「再連携後に再実行してください」       │
│  - 画面固有のリカバリ導線提供             │
└─────────────────────────────────────────┘
```

## カテゴリ A: 操作前に防ぐ（Guard / Disable / Status Check）

### A-1. 営業時間未設定でのシフト構築ブロック [Admin]

**シナリオ**: 管理者が営業時間を設定せずにビルダータブを開く → 日付クリックしても何も起きない / デフォルト値で動作

**対応**:
- ビルダータブ選択時に営業時間APIの結果をチェック
- 未設定ならバナー表示: 「営業時間が設定されていません」+ 「設定タブを開く」ボタン
- ビルダー操作エリアは表示するが、日付クリックを無効化

**受け入れ条件**:
- [ ] 営業時間が0件の状態でビルダータブを開くと、バナーが表示される
- [ ] バナーの「設定タブを開く」ボタンで設定タブに遷移する
- [ ] 営業時間設定後にビルダータブに戻るとバナーが消える

**対象ファイル**: `static/js/admin-app.js`
**イベント**: `builder_blocked_no_opening_hours`

### A-2. カレンダー未連携での同期操作ブロック [Admin]

**シナリオ**: Google Calendar未接続で「インポート」「エクスポート」「確定・同期」を押す → 401エラー

**対応**:
- 同期系ボタンの表示前に credentials 状態をチェック（missing / expired / revoked / partial scope）
- 未接続時: ボタンを disabled + インライン説明テキスト「Googleカレンダーが接続されていません」+ 「再ログインする」リンク
- expired 時: 説明テキスト「Googleの認証が切れています」+ 「再認証する」リンク

**受け入れ条件**:
- [ ] credentials なしで同期ボタンが disabled になる
- [ ] disabled ボタンの下に理由テキストと CTA が表示される
- [ ] 再ログイン後、ボタンが有効化される

**対象ファイル**: `static/js/admin-app.js`
**イベント**: `sync_blocked_no_credentials`

### A-3. closed/draft 期間での提出ブロック [Worker]

**シナリオ**: Worker が open 以外の期間を選択 → 提出ボタンを押す → 400エラー

**対応**:
- 期間選択時にステータスチェック
- `draft`: 提出ボタン非表示 + バナー「この期間はまだ募集開始されていません」
- `closed`: 提出ボタン非表示 + バナー「この期間の募集は終了しました」
- カレンダーは閲覧可能（既存提出の確認用）

**受け入れ条件**:
- [ ] draft 期間では提出ボタンが表示されない
- [ ] closed 期間では提出ボタンが表示されない
- [ ] 各状態に応じたバナーが表示される
- [ ] カレンダーグリッドは表示される（読み取り専用）

**対象ファイル**: `static/js/worker-app.js`
**イベント**: `submission_blocked_period_status`

### A-4. 二重送信防止 [Owner]

**シナリオ**: 承認ボタン連打 → 二重リクエスト

**対応**:
- 承認・差戻しボタンに `withLoading()` ラッパー適用
- 確認ダイアログ表示後〜API完了まで操作不可

**受け入れ条件**:
- [ ] 承認ボタンクリック後、ローディングスピナーが表示される
- [ ] ローディング中にボタンが disabled になる
- [ ] API完了後にボタンが復元される

**対象ファイル**: `static/js/owner-app.js`
**イベント**: なし（既存の approval_submitted で十分）

### A-5. メンバー操作の自己制約の説明 [Admin]

**シナリオ**: 自分のロール変更ドロップダウンが disabled → 理由不明

**対応**:
- disabled ドロップダウンの下にインラインテキスト: 「自分自身のロールは変更できません」
- 削除ボタンも同様: disabled + インラインテキスト「自分自身を除外することはできません」

**受け入れ条件**:
- [ ] 自分の行のロールドロップダウン下に説明テキストが表示される
- [ ] 自分の行の削除ボタンが disabled + 説明テキスト付きになる

**対象ファイル**: `static/js/admin-app.js`

## カテゴリ B: 失敗時に伝える（Error Mapping / Message / Feedback）

### B-1. api-client 共通エラーハンドリング強化

**対応**:
- エラーコード→日本語マッピングを追加（画面文脈なしの汎用メッセージ）:
  ```
  VALIDATION_ERROR → "入力内容に問題があります"
  NOT_FOUND → "データが見つかりません"
  INTERNAL_ERROR → "サーバーエラーが発生しました。時間をおいて再度お試しください"
  CREDENTIALS_EXPIRED → "Googleの認証が切れています。再ログインしてください"
  ORG_MEMBERSHIP_REQUIRED → "組織に参加していません"
  ```
- 429 専用ハンドラ: 「操作が多すぎます。しばらく待ってからやり直してください。」
- `TypeError`（ネットワークエラー）専用ハンドラ: 「ネットワークに接続できません」
- エラーオブジェクトに `code`, `message`, `status` を含めて throw → 画面層で文脈追加可能に

**受け入れ条件**:
- [ ] 429 レスポンスで専用メッセージが表示される
- [ ] ネットワーク切断時に専用メッセージが表示される
- [ ] エラーコードが定義済みの場合、日本語メッセージに変換される
- [ ] 未定義のエラーコードはバックエンドメッセージをフォールバック表示

**対象ファイル**: `static/js/modules/api-client.js`
**イベント**: `api_error_{code}` (code = error code)

### B-2. 認証切れモーダルの文脈メッセージ対応

**対応**:
- 共通 reauth モーダルに画面層からメッセージを注入できる仕組みを追加
- Worker 提出画面: 「編集中の内容は自動保存されています。再ログイン後に復元されます。」（C-1 のドラフト保存と連動）
- Admin 同期中: 「再認証後に同期を再実行してください。」
- デフォルト: 「再ログインしてください。」

**受け入れ条件**:
- [ ] Worker で 401 発生時、ドラフト保存済みなら復元可能メッセージが表示される
- [ ] Admin 同期操作中の 401 で再実行メッセージが表示される
- [ ] reauth モーダルにコンテキストメッセージ表示エリアがある

**対象ファイル**: `static/js/modules/api-client.js`, `static/js/worker-app.js`, `static/js/admin-app.js`

### B-3. カレンダーイベント取得失敗の通知 [Worker]

**シナリオ**: Google Calendar API エラーでイベント取得失敗 → console.warn のみ

**対応**:
- catch 内で warning トースト: 「カレンダーの予定を取得できませんでした。営業時間のみで計算しています。」
- カレンダーセレクター上にインラインバナー表示（一過性でなく残る）
- 「再取得」ボタン提供

**受け入れ条件**:
- [ ] カレンダー取得失敗時に warning トーストが表示される
- [ ] カレンダーセレクター上に警告バナーが残る
- [ ] 「再取得」ボタンで再試行できる

**対象ファイル**: `static/js/worker-app.js`
**イベント**: `calendar_fetch_failed`

### B-4. 確定時の部分同期失敗の可視化 [Admin]

**シナリオ**: 確定時に一部 Worker のカレンダー同期が失敗 → sync_summary は返るが詳細不明

**対応**:
- 確定後の結果をモーダルで表示（意思決定不要だがトーストでは情報量が足りない）:
  - 全成功: 成功トースト「5名のカレンダーに同期しました」
  - 部分失敗: 結果モーダル（失敗者リスト + 原因 + 対処法）
    - 例: 「山田太郎: 再認証が必要です → 本人に再ログインを依頼してください」
    - 例: 「佐藤花子: カレンダー権限不足 → 本人にカレンダー設定を確認するよう依頼してください」

**受け入れ条件**:
- [ ] 全員成功時はトーストのみ
- [ ] 1人以上失敗時はモーダルで失敗詳細が表示される
- [ ] 各失敗にエラー原因と推奨アクションが含まれる

**対象ファイル**: `static/js/admin-app.js`
**イベント**: `schedule_confirm_partial_failure`

### B-5. トースト通知の品質改善

**対応**:
- 最大同時表示数を3に制限（超過分は古いものから削除）
- エラートーストは自動消去しない（×ボタンで手動閉じ）
- 同一メッセージの重複防止（5秒以内に同じメッセージは表示しない）
- `role="alert"` と `aria-live="polite"` を追加

**受け入れ条件**:
- [ ] 4つ目のトーストが表示されると最古のトーストが消える
- [ ] エラートーストに×ボタンがあり、手動で閉じられる
- [ ] 同じメッセージが5秒以内に重複表示されない

**対象ファイル**: `static/js/modules/notification.js`

## カテゴリ C: 失敗後に回復させる（Draft / Retry / Partial Failure UI）

### C-1. Worker 提出データのドラフト自動保存

**シナリオ**: 認証切れ・ブラウザ閉じ・誤操作で編集中データが消失

**対応**:
- `slotData` + `notes` + `periodId` を `localStorage` に定期保存（30秒間隔 + 変更検知時）
- キー: `shift_draft_{periodId}`
- 再ログイン後/期間再選択時に自動復元 → info トースト「前回の編集内容を復元しました」
- 提出成功時にドラフト削除
- 復元データと現在の営業時間に矛盾がないかチェック（営業時間変更があれば警告）

**受け入れ条件**:
- [ ] 30秒ごとに slotData が localStorage に保存される
- [ ] ページリロード後に同じ期間を選択するとドラフトが復元される
- [ ] 復元時に info トーストが表示される
- [ ] 提出成功後にドラフトが削除される
- [ ] 営業時間が変更されていた場合に警告が表示される

**対象ファイル**: `static/js/worker-app.js`
**イベント**: `draft_restored`, `draft_conflict_detected`

### C-2. 未保存警告 [Worker]

**シナリオ**: シフト希望を編集中にブラウザバック/タブ閉じ → 編集内容消失

**対応**:
- `dirty` フラグ管理: slotData に変更があれば true
- `beforeunload` イベントで dirty 時に確認ダイアログ
- 期間リストに戻る操作にも確認: 「編集内容が保存されていません。戻りますか？」
- 提出成功で dirty リセット

**受け入れ条件**:
- [ ] 編集後にタブを閉じようとすると確認ダイアログが表示される
- [ ] 「戻る」操作で確認ダイアログが表示される
- [ ] 提出後にはダイアログが表示されない

**対象ファイル**: `static/js/worker-app.js`

### C-3. カレンダー再計算時の手動調整保護 [Worker]

**シナリオ**: カレンダー選択変更 → 全スロット再計算 → 手動調整済みの時間が上書きされる

**対応**:
- 再計算時に `is_custom_time === true` のスロットは自動計算で上書きしない
- カレンダー変更時に info トースト: 「カレンダー設定を変更しました。手動設定済みの日はそのまま維持されます。」
- 手動設定済みの日にも自動計算値を参考情報として表示（「自動計算: 10:00-18:00」）

**受け入れ条件**:
- [ ] カレンダー選択変更後、`is_custom_time` の日の時間が変わらない
- [ ] info トーストが表示される
- [ ] ポップアップ内に自動計算の参考値が表示される

**対象ファイル**: `static/js/worker-app.js`

## 実装順序

```
Step 1: 共通基盤（B-1, B-5, X-1）
  ├── api-client エラーハンドリング強化
  ├── トースト品質改善
  └── ダイアログ キーボード + ARIA

Step 2: 事故防止（A-1 ~ A-5）
  ├── 営業時間未設定ブロック
  ├── カレンダー未連携ブロック
  ├── 期間ステータス提出制御
  ├── Owner 二重送信防止
  └── メンバー自己制約説明

Step 3: データ喪失防止（C-1 ~ C-3）
  ├── Worker ドラフト保存
  ├── 未保存警告
  └── 手動調整保護

Step 4: エラー回復強化（B-2 ~ B-4）
  ├── 認証切れ文脈メッセージ
  ├── カレンダー取得失敗通知
  └── 部分同期失敗の可視化

Step 5: 学習導線（D-1 ~ D-3）
  ├── 初回チェックリスト
  ├── ヘルプボタン
  └── 空状態 CTA

Step 6: 仕様確認後（X-2）
  └── 割当超過警告
```

理由:
- Step 1 で共通基盤を固めることで、以降の実装が一貫する
- Step 2 は事故防止で最も実害を減らす
- Step 3 はデータ喪失防止で Worker 体験を大きく改善
- Step 4 はエラー回復で Step 1 の基盤上に構築
- Step 5 は学習導線で軽量実装から開始
- Step 6 は仕様確認が必要なため最後

## KPI & 計測

### 導入すべきイベントログ

| イベント | タイミング | 値 |
|---------|-----------|-----|
| `builder_blocked_no_opening_hours` | ビルダーで営業時間未設定検出時 | — |
| `sync_blocked_no_credentials` | 同期ボタン disabled 表示時 | credential_status |
| `submission_blocked_period_status` | 提出不可期間選択時 | period_status |
| `calendar_fetch_failed` | カレンダーイベント取得失敗時 | error_code |
| `draft_restored` | ドラフト復元時 | period_id |
| `draft_conflict_detected` | ドラフト復元時に営業時間変更検出 | period_id |
| `unsaved_warning_shown` | 未保存警告ダイアログ表示時 | — |
| `schedule_confirm_partial_failure` | 確定時の部分同期失敗 | failed_count, total_count |
| `onboarding_checklist_completed` | チェックリスト全完了 | — |
| `onboarding_checklist_dismissed` | チェックリスト手動非表示 | completed_items |
| `api_error_{code}` | API エラー発生時 | endpoint, status |

### 効果測定指標

| 指標 | 計測方法 | 目標 |
|------|---------|------|
| 400/401/429 発生件数 | サーバーログ集計 | 改善前比 50% 減 |
| Worker 提出完了率 | 期間オープン → 提出完了の割合 | 測定開始（現在値不明） |
| Admin 初回セットアップ完了率 | チェックリスト全完了率 | 80% 以上 |
| 確定時の同期失敗率 | sync_summary 集計 | 測定開始 |
| 同一操作の連打回数 | 429 発生 + ボタン連打検出 | 0 に近づける |

### 計測基盤

- フロントエンド: `navigator.sendBeacon('/api/analytics/event', ...)` で非同期送信
- バックエンド: 既存の AsyncTask + ログ基盤を活用
- **MVP**: まずはサーバーサイドの既存ログ（400/401/429 のステータスコード集計）で開始し、フロントエンドイベントログは Step 3 以降で導入
